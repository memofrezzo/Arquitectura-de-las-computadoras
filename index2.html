Módulo 01

¿Qué es Assembler?

Imaginá que querés darle instrucciones a alguien que solo entiende un idioma muy básico: solo entiende "encender" y "apagar" (1 y 0). Eso es lo que pasa con un microprocesador: solo entiende secuencias de ceros y unos.

Pensalo así: el procesador es como una persona que solo sabe leer código Morse. Si vos le escribís una carta en español, necesitás a alguien que la traduzca a Morse. Ese "traductor" es el ensamblador, y el "idioma intermedio" que vos usás para escribir de forma más legible (pero que todavía es bastante técnico) es el Assembler.

Set de Instrucciones: Es el conjunto de todas las instrucciones que una CPU puede ejecutar. Cada arquitectura de CPU tiene su propio set. El set de instrucciones también se conoce como lenguaje de máquina.

Una instrucción en lenguaje de máquina es una secuencia de bits (ceros y unos) que la CPU sabe interpretar. Por ejemplo:

Lenguaje de máquina 101110000000010100000000 101110000000001000000000 101000110000111000000000

Esas tres líneas de ceros y unos, juntas, simplemente suman 5 + 2 y guardan el resultado. Sí, todo eso solo para sumar dos números.

¿Por qué tantos ceros y unos para algo tan simple? Porque el procesador está construido para hacer operaciones extremadamente sencillas (mover un dato, sumar, comparar). Su poder viene de la velocidad: ejecuta millones de estas instrucciones por segundo. Combinando millones de operaciones simples, logra cosas complejas como mostrar un video o ejecutar un juego.

Regla fundamental: Cada instrucción de bajo nivel realiza una sola tarea básica.
Módulo 02

Lenguaje de Alto Nivel vs Bajo Nivel

Para no tener que escribir en ceros y unos, usamos lenguajes de alto nivel como C++, Java, Python, Ruby, etc. Estos lenguajes tienen instrucciones que los humanos podemos leer y escribir fácilmente.

Una sola instrucción de alto nivel agrupa varias instrucciones de bajo nivel:

Alto NivelBajo Nivel (lo que realmente hace el procesador)
resultado = 5 + 2 mov ax, 5 → Cargar 5 en AX
add ax, 2 → Sumarle 2 a AX
mov resultado, ax → Guardar el resultado

Es como la diferencia entre decir "haceme un café" (alto nivel) y tener que explicar paso a paso: "agarrá la taza, ponéla bajo la canilla, abrí la canilla, esperá a que se llene..." (bajo nivel). El lenguaje de alto nivel te permite expresar la intención; el de bajo nivel te obliga a detallar cada paso.

Módulo 03

Compiladores vs Ensambladores

Cuando escribís en un lenguaje de alto nivel, necesitás un compilador que traduzca tu código a lenguaje de máquina. Cada lenguaje tiene su propio compilador.

Compilador: Convierte código fuente de alto nivel en instrucciones de máquina de bajo nivel.
Ensamblador: Convierte código Assembler en código de máquina. A diferencia del compilador, no genera código nuevo: cada instrucción de Assembler se convierte en exactamente una instrucción de máquina.

La diferencia clave: el compilador "inventa" código. Vos le das una línea y él decide cuántas instrucciones de máquina necesita. El ensamblador es un traductor directo: una línea de Assembler = una instrucción de máquina, siempre. Esto te da control total sobre lo que hace el procesador.

El "problema" de los lenguajes de alto nivel es que, a cambio de facilidad, perdés control. El compilador decide por vos cómo traducir cada instrucción. Con Assembler, vos sabés exactamente qué va a ejecutar el procesador.

¿Para qué aprender Assembler hoy?

Porque te permite entender cómo funciona un programa compilado a nivel del microprocesador. Sabés que cada instrucción que escribís es una que el procesador va a ejecutar, y eso te da una comprensión profunda de lo que realmente hace tu código.

Por ejemplo: ¿sabías que para sumar 5 + 2 el procesador necesita tres instrucciones? En un lenguaje de alto nivel no se nota, pero en Assembler sí. Entender esto te convierte en un mejor programador, incluso cuando usás lenguajes de alto nivel.

Módulo 04

Arquitectura Intel 8086

Estudiamos Assembler basándonos en la arquitectura Intel 8086, que es la del microprocesador utilizado en la IBM PC original. A pesar de que la arquitectura ha evolucionado mucho, los principios generales se mantienen.

La arquitectura 8086 es como el "Modelo T" de los procesadores. Los autos modernos son mucho más avanzados, pero los principios básicos (motor, ruedas, volante) son los mismos. Aprender con el 8086 te da la base para entender cualquier procesador moderno.

Datos importantes de esta arquitectura:

  • Trabaja con datos de 8 bits (1 byte) y 16 bits (1 word/palabra)
  • Los registros internos son de 16 bits
  • El bus de direcciones es de 20 bits → puede direccionar hasta 1 MB de memoria
  • Utiliza memoria segmentada para resolver la diferencia entre registros de 16 bits y direcciones de 20 bits
Módulo 05

Herramientas y Creación de un .EXE

Herramientas necesarias

  • Editor: Para escribir el código fuente (cualquier editor de texto plano, como el Bloc de Notas)
  • Ensamblador (MASM): Traduce el programa fuente a lenguaje de máquina
  • Linker (LINK): Transforma el código de máquina en un ejecutable (.exe)

Proceso paso a paso

  1. Escribís tu código Assembler en un editor y lo guardás con extensión .asm
  2. Ejecutás MASM programa.asm; desde la línea de comandos. Esto genera programa.obj (código objeto: es código de máquina pero todavía no es ejecutable)
  3. Ejecutás LINK programa.obj; para generar el archivo programa.exe
programa.asmCódigo fuente
→ MASM →
programa.objCódigo objeto
→ LINK →
programa.exeEjecutable
Los programas MASM y LINK deben estar en la misma carpeta que el archivo .asm. Además, el nombre del archivo no debe superar los 8 caracteres (MASM y LINK no soportan nombres largos).
Módulo 06

Estructura de un Programa Assembler

Todo programa en Assembler tiene una estructura fija. La mayor parte es una "plantilla" estándar que siempre se repite:

Plantilla base .MODEL SMALL .STACK .DATA ; → Acá van las declaraciones de variables ← .CODE inicio: mov ax, @data ; Inicializar segmento de datos mov ds, ax ; → Acá va tu código ← mov ax, 4C00h ; Terminar el programa int 21h END inicio

Las dos secciones importantes

Segmento de datos (.DATA)

Aquí se declaran todas las variables que utiliza el programa. Es como la zona de "declarar variables" en un lenguaje de alto nivel.

Segmento de código (.CODE)

Aquí va el código del programa, es decir, las instrucciones que el procesador va a ejecutar. Siempre empieza con las líneas que inicializan el segmento de datos (mov ax, @data / mov ds, ax) y termina con las líneas que finalizan el programa (mov ax, 4C00h / int 21h).

mov ax, @data / mov ds, ax es como "encender" la zona de memoria donde están tus variables. Sin esto, el programa no puede encontrar tus datos. Las líneas mov ax, 4C00h / int 21h son como el "return" de un lenguaje de alto nivel: le dicen al sistema operativo que el programa terminó.

Ejemplo completo: sumar 5 + 2

Ejemplo .MODEL SMALL .STACK .DATA resultado DW ? ; Variable de 16 bits sin inicializar .CODE inicio: mov ax, @data mov ds, ax mov ax, 5 ; AX = 5 add ax, 2 ; AX = AX + 2 = 7 mov resultado, ax ; Guardar 7 en la variable resultado mov ax, 4C00h int 21h END inicio

Línea por línea: primero "prendemos" el acceso a las variables. Después cargamos el 5 en el registro AX (como si fuera una variable temporal). Luego le sumamos 2 a AX (ahora AX vale 7). Después copiamos ese valor a la variable "resultado" que vive en la memoria. Finalmente, le decimos al sistema operativo que terminamos.

Módulo 07

Registros del Intel 8086

Los registros son pequeñísimas áreas de memoria que están dentro del procesador mismo. Son súper rápidos de acceder porque no hay que ir a buscar nada a la memoria RAM.

Imaginá que estás cocinando. Los registros son como los bowls que tenés sobre la mesada: los usás para tener los ingredientes a mano mientras cocinás. La RAM sería la heladera: los ingredientes están ahí, pero tenés que ir a buscarlos cada vez que los necesitás. Usar registros es mucho más rápido que usar memoria.

Registros de propósito general

Son los registros que podés usar libremente para almacenar datos: AX, BX, CX y DX. Cada uno tiene 16 bits y se puede subdividir en dos partes de 8 bits:

Datos

Cada registro de 16 bits se puede usar como un todo (AX) o dividir en dos mitades de 8 bits (AH y AL). Las letras H y L vienen de "High" (alto) y "Low" (bajo). Es como tener una cajonera de 2 cajones: podés usar toda la cajonera o usar cada cajón por separado. Solo los registros de propósito general se pueden dividir así.

Ejemplo de uso

mov ax, 0057h ; AX = 0057h → AH=00h, AL=57h mov bh, 10101b ; BH = 00010101b (21 en decimal) mov bl, 3Bh ; BL = 3Bh

Módulo 08

Datos y Representación

No existen tipos de datos

En Assembler no existen los tipos de datos. La secuencia de bits 01100001 puede ser el número 97 o la letra "a", dependiendo de la operación que hagas sobre ella. Lo que importa es el tamaño.

En Assembler no hay tipos de datos. El significado de una secuencia de bits depende de la operación que se ejecute sobre ella.

Los dos tamaños posibles

TamañoNombreBits
8 bits1 bytexxxxxxxx
16 bits1 word (palabra)xxxxxxxx xxxxxxxx
No se pueden mezclar bytes con words en una misma instrucción. Si hacés MOV, ambos operandos deben ser de 8 bits o ambos de 16 bits.

Sistemas de numeración

Dentro del código Assembler los datos pueden expresarse en tres sistemas. Cómo identificar cuál es cuál:

SistemaIdentificaciónEjemplo
BinarioSe agrega b al final01111101b
HexadecimalSe agrega h al final000Fh
DecimalSe escribe normalmente7

El hexadecimal es muy usado porque cada dígito hexa representa exactamente 4 bits. Entonces 9E5Ch = 1001 1110 0101 1100b. Es mucho más corto y fácil de leer que el binario. Si un número hexa empieza con una letra (A-F), se le agrega un 0 adelante: A001h se escribe 0A001h para que el ensamblador no lo confunda con un nombre de variable.

Bits: más altos y más bajos

Los bits se numeran de derecha a izquierda empezando por el 0:

0 0 1 1 1 0 0 1
b7 b6 b5 b4 b3 b2 b1 b0
  • Bit 0 = bit menos significativo (LSB) — el más a la derecha
  • Bit 7 (o bit 15 para words) = bit más significativo (MSB) — el más a la izquierda

Una word (16 bits) se divide en dos bytes:

Byte alto (bits 15-8)Byte bajo (bits 7-0)
También llamado "byte más significativo"También llamado "byte menos significativo"
Módulo 09

Memoria y Direcciones

Organización de la memoria

La memoria es como una lista enorme de cajones numerados. Cada cajón guarda 1 byte (8 bits). Esa es la mínima unidad de memoria que se puede leer o escribir.

Mínima unidad de memoria direccionable: 8 bits (1 byte) — no se puede leer menos que eso.
Máxima unidad: 16 bits (2 bytes) — se pueden leer/escribir simultáneamente.

Direcciones de memoria

Cada cajón (celda) tiene un número identificador llamado dirección. Es como el número de casa en una calle: te permite encontrar un lugar específico.

0001h 0Ah
0002h F6h
0003h 14h
0004h 44h
... ...

En el Intel 8086, las direcciones son números de 20 bits. Eso permite direccionar 2²⁰ = 1.048.576 bytes = 1 MB de memoria. Pero como los registros solo tienen 16 bits (y solo pueden representar hasta 64 KB), se inventó un sistema especial llamado memoria segmentada.

Memoria segmentada

Las direcciones se expresan con dos números: segmento y desplazamiento.

SEGMENTO:DESPLAZAMIENTO

Es como las fechas del calendario: tenés el mes (1-12) y el día dentro del mes (1-31). De la misma forma, el segmento selecciona una porción de memoria y el desplazamiento indica un byte dentro de ese segmento.

La particularidad es que los segmentos se superponen: cada uno empieza 16 bytes después del anterior, pero puede abarcar 64 KB. Así, un mismo byte puede tener varias direcciones "nombres" distintas.

Direcciones segmentadas en el código

Para acceder a memoria se usa la sintaxis:

mov al, ds:[22] ; Copia el byte en la posición 22 del segmento de datos a AL

Formato Little Endian

Cuando se almacena un dato de 16 bits (una word) en memoria, se guarda primero la parte baja y luego la alta. Esto se llama formato little endian.

Por ejemplo, el número BE20h (que en binario es 10111110 00100000) se guarda así:

1000h 20h byte bajo
1001h BEh byte alto
El número queda "al revés" en memoria. Si lo leyéramos en orden (dirección 1000h primero), obtendríamos 20BEh, no BE20h. ¡Pero el procesador sabe cómo leerlo correctamente!
Módulo 10

Variables en Assembler

Una variable en Assembler es simplemente un nombre que se asigna a una dirección de memoria. El ensamblador se encarga de reemplazar el nombre por la dirección real.

Formato de declaración: NOMBRE TAMAÑO VALOR_INICIAL

DB = Define Byte (variable de 8 bits)
DW = Define Word (variable de 16 bits)

Ejemplos de declaración

.DATA NumeroPrimo DW 29 ; Reserva 2 bytes e inicializa con 29 Mes DB 6 ; Reserva 1 byte e inicializa con 6 Resultado DW ? ; Reserva 2 bytes SIN inicializar (contiene "basura")

NúmeroPrimo → Se reservan 2 bytes consecutivos y se escribe el 29 en formato little endian. En binario 16 bits, 29 = 00000000 00011101. Se guarda como 1Dh (bajo) y 00h (alto).

Mes → Se reserva 1 byte con el valor 6.

Resultado → La "?" significa: "solo reservame el espacio, no pongas ningún valor". La memoria queda con lo que sea que hubiera ahí antes (basura de otro programa).

Las variables quedan en orden en memoria

Las variables se almacenan en memoria en el mismo orden en que se declaran:

NumeroPrimo 1Dh
  00h
Mes 06h
Resultado ??
  ??

Usar variables en el código

mov ax, NumeroPrimo ; Copia el valor de NumeroPrimo en AX (AX = 29) mov bx, offset Resultado ; Copia la DIRECCIÓN de Resultado en BX (no el valor)

mov ax, NumeroPrimo → Accede al contenido de la variable (el valor 29).

mov bx, offset Resultado → Accede a la dirección de memoria donde vive la variable (no al valor que tiene adentro, sino a su "dirección postal").

Módulo 11

Arreglos

Un arreglo se declara poniendo varios valores separados por comas:

.DATA Pares DB 2, 4, 6, 8 ; Arreglo de 4 bytes

En memoria queda así:

Pares[0] 02h
Pares[1] 04h
Pares[2] 06h
Pares[3] 08h

Acceder a los elementos

mov ah, Pares ; AH = primer elemento = 2 (Pares[0]) mov cl, Pares[1] ; CL = segundo elemento = 4 mov dl, Pares[3] ; DL = cuarto elemento = 8

El número entre corchetes indica cuántos bytes avanzar desde el inicio del arreglo. Pares[1] significa "saltá 1 byte desde el inicio", o sea, andá al segundo elemento. Pares y Pares[0] son lo mismo: el primer elemento.

Arreglos de Words

Si el arreglo es de words (DW), cada elemento ocupa 2 bytes. Entonces el desplazamiento en bytes es el doble:

.DATA NumerosGrandes DW 1002h, 55A1h, 3037h, 000Fh ; Acceso: (desplazamiento en BYTES) ; Primer elemento: NumerosGrandes[0] ; Segundo elemento: NumerosGrandes[2] ← ¡avanza 2 bytes! ; Tercer elemento: NumerosGrandes[4] ; Cuarto elemento: NumerosGrandes[6]
El desplazamiento entre corchetes es siempre en bytes, sin importar si el arreglo es de bytes o de words. Para un arreglo de words, cada elemento está a 2 bytes de distancia.

Arreglos sin inicializar (DUP)

Arreglo1 DB 50 DUP (0) ; 50 bytes, todos inicializados en 0 Arreglo2 DB 50 DUP (?) ; 50 bytes, sin inicializar (basura) Arreglo3 DB 50 DUP (0Fh) ; 50 bytes, todos inicializados en 0Fh

DUP significa "duplicar". 50 DUP (0) dice: "repetí el valor 0, 50 veces". Es la forma de crear un arreglo grande sin tener que escribir cada valor a mano.

Módulo 13

MOV y XCHG

MOV — Instrucción de asignación

Es la instrucción más básica: copia un valor de un lugar a otro. Equivale al signo = en los lenguajes de alto nivel.

MOV Destino, Fuente ; Destino = Fuente mov ax, 000Fh ; AX = 000Fh (hexadecimal) mov cx, ax ; CX = AX (copia el valor de AX en CX) mov bh, 7 ; BH = 7 (decimal) mov dl, 01111101b ; DL = 01111101b (binario)

mov ax, 5 se lee como: "mové el 5 a AX", o mejor: "AX = 5". No confundir con "move" en el sentido de que el origen se borre: MOV copia, no mueve. El valor original queda intacto.

Regla importante: El destino (primer operando) nunca puede ser un número. Tiene que ser un registro o una dirección de memoria. No tiene sentido "escribir en el número 5".

XCHG — Intercambio

Intercambia los valores de dos operandos:

mov ax, 1 ; AX = 1 mov bx, 2 ; BX = 2 xchg ax, bx ; Ahora AX = 2 y BX = 1
Ninguno de los operandos de XCHG puede ser un número (constante). Ambos deben ser registros o direcciones de memoria, porque el procesador necesita un lugar físico donde guardar cada valor.
Módulo 14

Instrucciones Aritméticas Básicas

ADD — Suma

Suma dos operandos y guarda el resultado en el primer operando (el destino).

ADD Destino, Operando2 ; Destino = Destino + Operando2 mov ah, 1 ; AH = 1 mov bh, 2 ; BH = 2 add ah, bh ; AH = AH + BH → AH = 3
En todas las instrucciones que almacenan un resultado, el resultado se guarda en el primer operando.

INC — Incrementar en 1

INC Destino ; Destino = Destino + 1 (más rápida que ADD Destino, 1) mov dx, 6 ; DX = 6 inc dx ; DX = 7 inc dx ; DX = 8

SUB — Resta

SUB Destino, Operando2 ; Destino = Destino - Operando2 mov dh, 10 ; DH = 10 mov bl, 2 ; BL = 2 sub dh, bl ; DH = DH - BL → DH = 8
La resta es siempre en este orden: Destino - Operando2, no al revés.

DEC — Decrementar en 1

DEC Destino ; Destino = Destino - 1 (más rápida que SUB Destino, 1) mov dx, 6 ; DX = 6 dec dx ; DX = 5 dec dx ; DX = 4

Ejemplo completo: resolver (355 + 281 - 1) - (413 - 78)

.MODEL SMALL .STACK .DATA .CODE inicio: mov ax, @data mov ds, ax ; Resolver primer paréntesis → queda en DX mov dx, 355 ; DX = 355 add dx, 281 ; DX = 355 + 281 = 636 dec dx ; DX = 636 - 1 = 635 ; Resolver segundo paréntesis → queda en AX mov ax, 413 ; AX = 413 sub ax, 78 ; AX = 413 - 78 = 335 ; Restar los resultados de ambos paréntesis sub dx, ax ; DX = 635 - 335 = 300 mov ax, 4C00h int 21h END inicio

La estrategia es: usá un registro para cada parte de la operación. Primero resolvé cada paréntesis por separado (usando registros distintos), y después combiná los resultados. Es como resolver una cuenta larga en papel: primero resolvés cada parte y después juntás todo.

Módulo 15

Multiplicación y División

MUL — Multiplicación sin signo / IMUL — Con signo

MUL/IMUL solo usa un operando. El otro factor siempre viene de un registro específico:

Multiplicar dos bytes (8 bits)

mov al, 5 ; Un factor va en AL mov bl, 2 ; El otro factor mul bl ; AX = AL * BL = 5 * 2 = 10

¿Por qué el resultado va en AX (16 bits) y no en AL (8 bits)? Porque multiplicar dos números de 8 bits puede dar hasta 255 × 255 = 65.025, ¡y eso no entra en un byte! Por eso el resultado se guarda en AX (16 bits).

Multiplicar dos words (16 bits)

mov ax, 1000 ; Un factor va en AX mov bx, 3000 ; El otro factor mul bx ; Resultado en DX:AX (32 bits) ; DX = parte alta del resultado ; AX = parte baja del resultado

Multiplicar dos números de 16 bits puede dar hasta 65.535 × 65.535 = ¡más de 4 mil millones! Eso no entra en 16 bits (máximo 65.535), así que el resultado se divide en dos registros: DX tiene la parte alta y AX la parte baja.

DIV — División sin signo / IDIV — Con signo

Dividir un número de 16 bits por uno de 8 bits

mov ax, 120 ; Dividendo en AX mov bl, 2 ; Divisor div bl ; AL = cociente (60), AH = resto (0)

Dividir un número de 32 bits por uno de 16 bits

; Dividendo en DX:AX mov dx, 0 ; Parte alta del dividendo mov ax, 1000 ; Parte baja del dividendo mov bx, 3 ; Divisor div bx ; AX = cociente, DX = resto
Resumen de dónde van los operandos:
MUL byte: AL × operando → resultado en AX
MUL word: AX × operando → resultado en DX:AX
DIV byte: AX / operando → cociente en AL, resto en AH
DIV word: DX:AX / operando → cociente en AX, resto en DX
Módulo 16

Operaciones Lógicas

Estas operaciones trabajan bit a bit (cada bit se compara con el bit correspondiente del otro operando).

InstrucciónOperaciónEfecto
ANDConjunción (Y lógico)Resultado = 1 solo si AMBOS bits son 1
ORDisyunción (O lógico)Resultado = 1 si AL MENOS UN bit es 1
XORDisyunción exclusivaResultado = 1 si los bits son DISTINTOS
NOTNegaciónInvierte cada bit (0→1, 1→0) = complemento a 1
NEGComplemento a 2Genera el negativo del número (complemento a 2)
TESTAND sin guardarComo AND pero NO guarda el resultado; solo afecta flags

Ejemplos

; AND - ambos bits deben ser 1 mov al, 01001110b mov ah, 00111001b and al, ah ; AL = 00001000b ; OR - al menos un bit debe ser 1 mov al, 01001110b mov ah, 00111001b or al, ah ; AL = 01111111b ; XOR - bits deben ser distintos mov al, 01001110b mov ah, 00111001b xor al, ah ; AL = 01110111b ; NOT - invierte todos los bits mov al, 01001110b not al ; AL = 10110001b ; NEG - complemento a 2 (niega el número) mov al, 01001110b neg al ; AL = 10110010b

AND: Usálo como "filtro" — solo deja pasar los bits que están en 1 en ambos operandos. Útil para extraer ciertos bits de un byte.

OR: Usálo para "encender" bits específicos sin tocar los demás.

XOR: Útil para invertir bits seleccionados. También: XOR reg, reg es la forma más rápida de poner un registro en cero.

NOT vs NEG: NOT invierte bits (complemento a 1). NEG invierte bits Y le suma 1 (complemento a 2, que es la representación del negativo).

Módulo 17

Modos de Direccionamiento

Los modos de direccionamiento son las distintas formas de indicar dónde están los datos que una instrucción necesita.

Reglas generales

Tipos de operandos posibles en MOV:
MOV Registro, Registro
MOV Registro, Constante
MOV Registro, Dirección de memoria
MOV Dirección de memoria, Registro
MOV Dirección de memoria, Constante

No existe: MOV Dirección, Dirección (no se pueden usar dos direcciones de memoria)

Si querés copiar el contenido de una dirección de memoria a otra dirección, tenés que pasar por un registro intermedio. Es como si no pudieras transferir dinero directamente de una cuenta bancaria a otra del mismo banco: tenés que sacarlo primero y después depositarlo.

1. Direccionamiento Inmediato

El dato es una constante escrita directamente en la instrucción:

add ax, 50 ; El 50 es un dato inmediato mov bh, 7 ; El 7 es un dato inmediato

2. Direccionamiento Inherente (de Registro)

El dato está en un registro:

mov ax, bx ; Copia el contenido de BX en AX add ah, al ; Suma AL a AH

3. Direccionamiento Directo

El dato está en memoria y la dirección se escribe explícitamente (o se usa una variable):

mov ax, ds:[107Ah] ; Lee de la dirección 107Ah mov bx, NumeroPrimo ; Lee de la dirección de NumeroPrimo sub ch, arreglo[7] ; Lee del arreglo en el byte 7

4. Direccionamiento Indirecto por Registro

El dato está en memoria pero la dirección está guardada en un registro:

mov ax, ds:[bx] ; Lee de la dirección que está en BX

Es como tener la dirección de una casa guardada en un papel. BX contiene la dirección, y entre corchetes le decimos al procesador "andá a buscar el dato que está en esa dirección". Los registros que pueden usarse así son: BX, BP, SI y DI.

5. Direccionamiento Relativo

La dirección se calcula con una suma de registro(s) + constante:

mov ax, ds:[bx + si + 5] ; Dirección = BX + SI + 5 mov ax, arreglo[bx] ; Dirección = dirección de arreglo + BX

Este modo es súper útil con arreglos. Suponé que tenés un arreglo que empieza en la dirección 1000h, y BX vale 3. Entonces arreglo[BX] accede a la posición 1000h + 3 = 1003h. Es como decir "andá al arreglo y avanzá BX posiciones".

Los registros base son BX y BP. Los índice son SI y DI.

Módulo 18

Control de Flujo

¿Cómo ejecuta las instrucciones el procesador?

La CPU ejecuta un ciclo de tres fases:

  1. Fetch (Búsqueda): La CPU busca la siguiente instrucción en el segmento de código. El registro IP (Instruction Pointer) le dice dónde está.
  2. Decode (Decodificación): La CPU interpreta qué tiene que hacer con esa instrucción.
  3. Execute (Ejecución): La CPU realiza la operación y guarda el resultado.

Normalmente las instrucciones se ejecutan una tras otra (de arriba hacia abajo). El registro IP se incrementa automáticamente para apuntar a la siguiente. Pero a veces necesitamos "saltar" a otra parte del código — como un goto o un if. Para eso están las instrucciones de control de flujo.

JMP — Salto incondicional

Salta siempre a la etiqueta indicada, sin importar nada:

inicio: ; ... código ... jmp inicio ; Siempre vuelve a "inicio" (¡ciclo infinito!)

CMP — Comparación

Compara dos valores. No guarda ningún resultado, pero modifica el registro de flags para que los saltos condicionales sepan qué hacer.

CMP Operando1, Operando2 ; Compara (internamente hace Operando1 - Operando2)

Saltos condicionales

Saltan a una etiqueta solo si se cumple una condición:

InstrucciónCondiciónTipo
JESaltar si son igualesCualquiera
JNESaltar si no son igualesCualquiera
JASaltar si es mayor (sin signo)BSS
JAESaltar si es mayor o igualBSS
JBSaltar si es menor (sin signo)BSS
JBESaltar si es menor o igualBSS
JGSaltar si es mayor (con signo)Ca2
JGESaltar si es mayor o igualCa2
JLSaltar si es menor (con signo)Ca2
JLESaltar si es menor o igualCa2

BSS = Binario sin signo (solo positivos: 0 a 255 en byte). Ca2 = Complemento a 2 (con signo: -128 a +127 en byte). Usá JA/JB cuando sabés que los números son siempre positivos, y JG/JL cuando pueden ser negativos.

Estructura IF...ELSE

Para hacer un IF en Assembler, la lógica es invertida: saltás el bloque "if" cuando la condición NO se cumple.

; En alto nivel: ; if (AX == BX) { hacer_algo } else { hacer_otro } cmp ax, bx jne else ; Si NO son iguales, salta al else ; Código cuando AX == BX jmp fin ; Salta el bloque else else: ; Código cuando AX != BX fin: ; Continúa el programa

¿Por qué la condición está invertida? Porque el IF de alto nivel dice "si la condición se CUMPLE, ejecutá el bloque". En Assembler, el salto condicional dice "si la condición se cumple, SALTEÁ el bloque". Son opuestos: si el IF dice "si son iguales", el salto dice "si NO son iguales, salteá". Por eso JE se convierte en JNE y viceversa.

Módulo 19

Iteraciones

LOOP — Iteración fija (tipo FOR)

LOOP usa el registro CX como contador. Cada vez que se ejecuta LOOP, CX se decrementa en 1 y salta a la etiqueta si CX no es cero.

mov cx, 50 ; Iterar 50 veces mov si, 0 ; Empezar por el elemento 0 siguiente: mov arreglo[si], 0 ; Poner un 0 en la posición actual inc si ; Ir al próximo elemento loop siguiente ; CX--, si CX != 0 volver a "siguiente"

Es como decir "hacé esto 50 veces". CX es el contador que empieza en 50 y baja de a 1. Cuando llega a 0, el loop termina. Es equivalente a: for (int i = 50; i > 0; i--) { ... }

Iteración condicional (tipo WHILE)

Cuando necesitás iterar mientras se cumpla una condición (no una cantidad fija):

mov si, 0 ; Empezar por el elemento 0 siguiente: cmp arreglo[si], 0FFh ; ¿El elemento actual es FFh? je salir ; Si es FFh, salir del bucle mov arreglo[si], 0 ; Poner un 0 en la posición actual inc si ; Ir al próximo elemento jmp siguiente ; Volver a iterar salir:

Este código recorre un arreglo byte por byte, poniendo 0 en cada posición, hasta encontrar un byte con valor FFh (que actúa como "señal de fin"). Es como decir: "mientras el elemento no sea FFh, poné cero y avanzá al siguiente".

Ejemplo: determinar longitud de un arreglo terminado en 00h

.DATA arreglo DB 15h, 07h, 22h, 9Ah, 4Dh, 00h .CODE inicio: mov ax, @data mov ds, ax mov dx, 0 ; Longitud = 0 (contador) mov si, 0 ; Índice = 0 iterar: cmp arreglo[si], 00h ; ¿Llegamos al final (00h)? je fin ; Si sí, terminar inc dx ; Incrementar longitud inc si ; Ir al próximo byte jmp iterar ; Repetir fin: mov ax, 4C00h int 21h

Recorremos el arreglo posición por posición. En cada paso preguntamos: "¿este byte es 00h?". Si no lo es, incrementamos el contador de longitud (DX) y avanzamos al siguiente byte. Cuando encontramos el 00h, salimos del bucle. Al final, DX = 5 (la longitud sin contar el 00h final).

Módulo 20

Números Negativos: Complemento a 2

Los números negativos se representan en complemento a 2 (Ca2). La secuencia de bits es la misma; depende de si usás instrucciones sin signo (BSS) o con signo (Ca2) para interpretarla.

Regla del complemento a 2 para negar un número:
1. Invertir todos los bits (complemento a 1 = NOT)
2. Sumar 1

Ejemplo: -2 en Ca2 de 8 bits:
2 = 00000010 → NOT = 11111101 → +1 = 11111110 → -2 = 11111110b

Identificar si un número es negativo

Es muy fácil: mirá el bit más significativo (MSB).

  • Si MSB = 0 → positivo
  • Si MSB = 1 → negativo (en Ca2)

Para números hexadecimales, fijate si empieza con un dígito entre 8 y F. Si es así, es negativo en Ca2. Ejemplos: FFh = negativo, 7Fh = positivo, A0h = negativo, 30h = positivo.

Usar el signo menos delante de un número al declarar una variable no garantiza que el resultado sea negativo. El signo menos simplemente aplica la regla de complemento a 2. Si el resultado no entra en el tamaño del dato, se produce un desborde y el número puede terminar siendo positivo.

Ejemplo: numero DB -195 → el resultado es 61 (positivo) porque -195 no entra en un byte con signo.
Módulo 21

Registro de Flags

El registro de flags (banderas) es un registro de 16 bits que la CPU modifica automáticamente después de cada operación aritmética o lógica. Cada bit indica una "condición" del resultado.

Pensalo como las luces de advertencia del tablero de un auto: después de cada operación, la CPU "enciende" ciertas luces (pone bits en 1) para indicar qué pasó con el resultado. Los saltos condicionales (JE, JNE, JA, etc.) leen estas luces para decidir si saltan o no.

Los flags más importantes

FlagNombreSe activa cuando...
ZZero (Cero)El resultado de la operación es 0
SSign (Signo)El resultado es negativo (MSB = 1)
CCarry (Acarreo)Hay un acarreo fuera del MSB en operaciones sin signo (BSS)
OOverflow (Desborde)Hay un desborde aritmético en complemento a 2

Ejemplo

mov ax, 5 mov bx, 7 sub ax, bx ; 5 - 7 = -2 = 11111110b ; Z = 0 (el resultado no es cero) ; S = 1 (el MSB es 1, resultado negativo en Ca2) ; C = 1 (hubo acarreo/borrow) ; O = 0 (no hubo desborde en Ca2)

Los saltos condicionales leen estos flags. Por ejemplo, JE salta si Z=1 (el resultado fue cero). JL (saltar si menor, con signo) revisa si S y O tienen ciertos valores. No necesitás manipular los flags directamente: la CPU lo hace automáticamente.

Módulo 22

Interrupciones

Una interrupción es un mecanismo que detiene momentáneamente el programa en ejecución para atender un evento especial.

Imaginá que estás estudiando y suena el teléfono. Pausás lo que estás hacés, atendés la llamada, y cuando terminás volvés exactamente a donde dejaste. Eso es una interrupción: el procesador pausa el programa, atiende el evento, y después continúa.

Tipos de interrupciones

Interrupciones por hardware

Las genera el hardware cuando ocurre un evento físico (el usuario presiona una tecla, llega un dato del disco, etc.).

  • Cada dispositivo tiene una línea de solicitud de interrupción (IRQ)
  • Cuando el evento ocurre, la CPU detiene el programa y ejecuta la subrutina de atención (ISR)
  • Al terminar la ISR, el programa continúa donde se quedó

El teclado, por ejemplo, usa IRQ1. Cada vez que presionás una tecla, se envía una señal por IRQ1, la CPU pausa lo que hace, lee la tecla, la guarda en un buffer (una memoria temporal), y vuelve al programa. Por eso no se pierden las pulsaciones aunque el procesador esté ocupado con otra cosa.

Interrupciones por software

Son subrutinas especiales del BIOS o del sistema operativo que se invocan con la instrucción INT. La más importante para nosotros es INT 21h (servicios del DOS).

Servicios del DOS (INT 21h)

Se establece un código de función en AH y se ejecuta INT 21h:

AHFunciónParámetrosResultado
01h Leer un carácter del teclado — AL = código ASCII del carácter
02h Mostrar un carácter en pantalla DL = carácter a mostrar —
09h Mostrar un string en pantalla DX = dirección del string (terminado en "$") —
0Ah Leer un string desde teclado DX = dirección del buffer String almacenado en el buffer
4Ch Terminar el programa AL = código de retorno (00h = OK) Vuelve al sistema operativo

Ejemplo: mostrar "Hola" en pantalla

.DATA mensaje DB "Hola$" ; DEBE terminar con "$" .CODE mov ah, 09h ; Función: mostrar string mov dx, offset mensaje ; Dirección del string int 21h ; Ejecutar

Primero le decimos a DOS qué queremos hacer (función 09h = mostrar texto). Después le indicamos dónde está el texto (la dirección de memoria donde empieza "Hola$"). Finalmente, int 21h ejecuta la operación. El string DEBE terminar con el carácter "$" para que DOS sepa dónde termina.

Ejemplo: leer un carácter del teclado

mov ah, 01h ; Función: leer carácter int 21h ; Espera a que el usuario teclee algo ; Ahora AL tiene el código ASCII del carácter

Leer un string desde teclado

Para leer un string se necesita un buffer con un formato especial:

OffsetContenido
0Longitud máxima de caracteres (incluyendo Enter)
1Se llenará con la cantidad de caracteres realmente leídos
2 en adelanteAquí se almacena el string leído
.DATA buffer DB 5, 0, 5 DUP (0) ; Máx 5 chars, espacio para almacenarlos .CODE mov ah, 0Ah ; Función: leer string mov dx, offset buffer ; Dirección del buffer int 21h ; El string queda guardado a partir del offset 2 del buffer ; La cantidad de caracteres leídos queda en el offset 1

El buffer tiene un formato especial que DOS espera: primero le decimos cuántos caracteres como máximo puede leer (offset 0). DOS nos devuelve cuántos realmente leyó (offset 1). Y a partir del offset 2 queda el texto tecleado. Es como darle a alguien un formulario con el espacio máximo disponible y que te lo devuelva con la cantidad de casillas que realmente llenó.
